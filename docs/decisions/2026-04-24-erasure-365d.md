# C.L — Erasure 365d (derecho al olvido estructurado)

**Fecha**: 2026-04-24
**Status**: Implementado
**Sub-milestone**: C.L

## Contexto

El principio canónico (`CLAUDE.md § Sobre la identidad de los miembros`)
establece:

> "Derecho al olvido estructurado. El contenido que alguien crea queda en
> el place; su rastro personal se borra al salir; su nombre se desliga del
> contenido tras 365 días."

La ontología (`docs/ontologia/miembros.md § Cuatro`) detalla:

- **Al salir** (`Membership.leftAt`): rastro personal (presencia,
  actividad, lecturas) se borra inmediatamente.
- **A los 365 días**: pasa a "ex-miembro", su nombre se desliga del
  contenido. El contenido (posts, comments) persiste.

Al planificar C.L descubrimos que **gran parte del contrato ya está
implementado funcionalmente**:

- `Post.authorUserId` y `Comment.authorUserId` nullable con
  `onDelete: SetNull` ✅
- `authorSnapshot Json NOT NULL` congelado al crear ✅
- RLS via `is_active_member()` invisibiliza al ex-miembro en todas las
  queries SELECT/INSERT/UPDATE ✅
- `listReadersByPost` filtra ex-miembros ✅
- Reactions retornan counts agregados sin userId ✅

El único gap: **no existe el job que a los 365d nullifique
`authorUserId` y renombre `authorSnapshot.displayName → "ex-miembro"`**.

## Decisiones tomadas

### 1. Interpretación funcional del contrato (no borrado físico)

**Decisión**: NO hace falta `DELETE` físico de `Reaction` / `PostRead` /
`Flag.reporterUserId`. El contrato "se borra" se cumple funcionalmente
porque:

- RLS bloquea INSERT/SELECT/UPDATE del ex-miembro (`is_active_member`
  retorna false cuando `leftAt IS NOT NULL`).
- `listReadersByPost` filtra por `Membership` activa → el lector ex-
  miembro no aparece en el bloque "Leyeron".
- Reactions son counts agregados — el user ex-miembro no es
  identificable individualmente.
- `Flag.reporterUserId` solo visible al admin (trazabilidad deseada).

**Alternativa descartada (borrado físico)**: cambiar FKs `Restrict →
SetNull` en Reaction/PostRead/Flag + DELETE explícitos en
`leaveMembershipAction`. No agrega privacidad (RLS ya los invisibiliza)
y complica el schema innecesariamente.

### 2. Plataforma: Vercel Cron

**Decisión**: `GET /api/cron/erasure` + `vercel.json` con schedule
diario. Segundo cron `GET /api/cron/erasure-audit` semanal para detectar
fallas silenciosas.

**Rationale**:

- Fit natural al stack (Next+Vercel).
- Cero infra nueva.
- Logs en Vercel Functions dashboard.
- Gratis en tier Hobby hasta 2 crons (cumplimos exacto).

**Alternativas descartadas**:

- **pg_cron (Supabase)**: precedente nuevo; agrega SQL function que
  nadie más del repo usa. Mejor reservar para casos que requieran
  atomicidad estricta dentro de la DB.
- **GitHub Actions schedule**: depende de GH Actions uptime. Overhead
  de workflow extra.

**Particularidades de Vercel Cron documentadas** (para futuros crons):

- Usa **GET, no POST** — error común que romperíamos si usáramos POST.
- **NO reintenta** ante 5xx — el cron audit semanal es la salvaguarda.
- **Puede duplicar eventos** — idempotencia obligatoria + advisory lock.
- Header `Authorization: Bearer <CRON_SECRET>` inyectado
  automáticamente por Vercel si el secret está en env vars.
- `maxDuration=300` declarado en el route (Pro tier); `runtime='nodejs'`
  explícito para Prisma.

### 3. Audit trail: tabla `ErasureAuditLog` con `snapshotsBefore`

**Decisión**: cada run del job inserta 1 fila por membership procesada,
capturando los snapshots ANTES del UPDATE.

**Rationale**: permite **rollback manual via SQL** si se detecta:

- Bug en `leftAt` setting (ej: fecha corrompida).
- Ejecución prematura por clock skew.
- Disputa legal del ex-miembro.

Sin este audit, el `displayName` original quedaría overwritten
irrecuperablemente. Backup de Supabase solo dura 7-28 días (tier
dependent) — no cubre la ventana de 365d.

**Rollback runbook** (SQL admin):

```sql
-- Encontrar la entry de audit
SELECT * FROM "ErasureAuditLog" WHERE "membershipId" = '...';

-- Recuperar snapshots originales
UPDATE "Post" p
SET
  "authorUserId" = audit."userId",
  "authorSnapshot" = jsonb_build_object(
    'displayName', entry->>'displayName',
    'avatarUrl', entry->>'avatarUrl'
  )
FROM "ErasureAuditLog" audit,
  jsonb_array_elements(audit."snapshotsBefore") entry
WHERE audit.id = '<audit-id>'
  AND entry->>'type' = 'POST'
  AND p.id = entry->>'id';

-- Ídem Comment
-- Resetear membership
UPDATE "Membership" SET "erasureAppliedAt" = NULL WHERE id = '<membership-id>';
```

### 4. Dry-run mode via `?dryRun=true`

**Decisión**: el route acepta query param. En `dryRun=true`:

- Captura snapshots en `ErasureAuditLog` con `dryRun=true`.
- **No** ejecuta UPDATE Post/Comment.
- **No** marca `Membership.erasureAppliedAt`.
- La tx rollbackea (via `DryRunAbort` interno) → el insert de audit NO
  se commitea tampoco. El counter del result refleja cuántos se
  HABRÍAN procesado.

**Valor**: primer run en producción puede ser dry-run para verificar
eligibility sin aplicar. Agendar `?dryRun=true` antes del deploy real.

### 5. Safety threshold: `leftAt > now() - 10 años`

**Decisión**: el job query tiene `leftAt: { lt: cutoff, gt: safetyFloor }`
donde `safetyFloor = now() - 10 años`.

**Rationale**: si alguien ejecuta manualmente `UPDATE Membership SET
leftAt = now() - '400 days'` por bug, el job no lo acepta como "1 año
viejo" — el filtro de safety catches fechas anómalas. Defense-in-depth.

### 6. Skipear places archivados

**Decisión**: el WHERE del job incluye `AND place.archivedAt IS NULL`.

**Rationale** (decisión user-approved): un place archivado tiene
contenido invisible a todos los members (incluido ex-miembros). No
justifica gastar computación procesando content que nadie ve. Si el
place se desarchiva en el futuro (hoy archivado es monótono), los
ex-miembros podrían quedar expuestos — aceptamos ese trade-off.

### 7. Tx única per-membership (all-or-nothing)

**Decisión**: `processOneMembership` envuelve Post UPDATE + Comment
UPDATE + ErasureAuditLog INSERT + Membership UPDATE en una sola
`prisma.$transaction`. Si falla alguno, rollback completo, membership no
se marca, reintenta al día siguiente.

**Rationale**: garantiza consistencia sin locks globales. Error en una
membership no aborta las demás (error isolation en el loop).

### 8. Advisory lock Postgres

**Decisión**: `pg_try_advisory_lock(36524)` al inicio del handler. Si
otro worker está corriendo, retorna noop con log warn.

**Rationale**: Vercel Cron puede duplicar eventos (documented
behavior). Sin el lock, dos workers podrían leer las mismas
memberships eligibles y duplicar counts (inofensivo pero gastador).
Advisory lock es barato y correcto.

### 9. Prisma directo bypassa RLS

**Decisión**: `runErasure` usa `prisma.*` directo (DATABASE_URL del
pooler) que pasa como `postgres` role → bypassa RLS automáticamente.

**Rationale**: el job necesita UPDATE sobre Post/Comment con
`authorUserId NULL` como resultado. La RLS de Post/Comment tiene
policies de UPDATE que asumen `authorUserId = auth.uid()::text` —
con null, esas policies fallarían. Bypasear RLS para jobs
administrativos es pattern estándar de Supabase (service_role o
Prisma con connection pooler sin JWT).

## Runbook: rotación de `CRON_SECRET`

Rotación anual o bajo sospecha de leak:

1. Generar nuevo secret: `openssl rand -hex 32`.
2. Agregar `CRON_SECRET_PREVIOUS` a env Vercel con el valor actual.
3. Actualizar el handler `isAuthorized` para aceptar ambos secrets
   durante 7 días de gracia (cambio de código, deploy).
4. Rotar `CRON_SECRET` en Vercel env → nuevo valor.
5. Siguiente cron run usa el nuevo secret automáticamente.
6. Tras 7 días: remover `CRON_SECRET_PREVIOUS` + revertir handler.

Sin período de gracia, la rotación causa downtime entre el momento del
deploy y el update del env var. 7 días es margen conservador.

## Observability

Eventos pino estructurados:

- `erasureRun` (info): un run completo (real o dry-run) con counts.
- `erasureMembershipFailed` (error): una tx de membership falló. Las
  demás continúan.
- `erasureLockContention` (warn): advisory lock ocupado (otro worker).
- `cronErasureFailed` (error): el handler HTTP tiró 500.
- `erasureBacklog` (warn): cron audit detectó memberships elegibles sin
  procesar.
- `erasureAuditClean` (info): cron audit sin backlog.

Admin revisa Vercel Functions logs filtrando por estos eventos.

## Alternativas descartadas

- **UI admin `/settings/erasure`** para revisar audit log + rollback:
  diferido. Hoy admin accede via SQL Supabase dashboard.
- **Endpoint `/api/admin/erasure/rollback`**: requiere infra de 2FA que
  no existe. Rollback via SQL documentado.
- **Cleanup at-leave** (capa 1 literal): interpretación funcional vía
  RLS + filters.
- **Cambio de FKs Restrict → SetNull** en Reaction/PostRead/Flag:
  innecesario.
- **Retro-anonimización de `quotedSnapshot.authorLabel`** en comments
  que citan a ex-miembros: asimetría histórica intencional — el
  snapshot se congela al momento de citar.

## Out of scope (post-MVP)

- Broadcast realtime del rename (notificar a users con WebSocket
  abierto): aceptable que el cambio aparezca al siguiente navigate.
- Test automatizado de rotación de `CRON_SECRET`: runbook manual.
- Procesar places archivados (decisión explícita).
- Endpoint UI de rollback.

## Verificación

### Automatizada

- 9 tests unit del job (eligibility, tx, error isolation, advisory
  lock, dry-run, snapshot shape, safety threshold).
- 7 tests del endpoint erasure (gate 401, dryRun param, 500 handling,
  timing-safe compare).
- 4 tests del endpoint erasure-audit (gate, backlog > 0, backlog = 0,
  query shape).

### Manual post-deploy (staging primero)

1. Env `CRON_SECRET` configurado en Vercel (32+ chars).
2. Dry-run first:
   `curl -H "Authorization: Bearer $CRON_SECRET" https://staging/api/cron/erasure?dryRun=true`.
3. Verificar entry en `ErasureAuditLog` con `dryRun=true`.
4. Crear test user con `leftAt` artificial 400d atrás, ejecutar real
   run, verificar DB + UI.
5. Audit semanal: `/api/cron/erasure-audit` → backlog 0.
