# Gotchas

Comportamientos que (a) no son derivables del código, (b) tienen síntoma confuso, (c) volverían a morder. Criterio en `CLAUDE.md` § Gotchas.

- [Cookies `__Secure-` de Neon Auth requieren HTTPS](neon-auth-secure-cookie-https.md) — sesiones no persisten en dev local sobre http plano.
- [El driver `@neondatabase/serverless` no parsea uniformemente los arrays](neon-serverless-array-parsing.md) — `array_agg`/`text[]` vuelve como literal `'{a,b}'`, no array JS; usar `string_agg`+split en introspección/runtime.
- [next-intl tira `FORMATTING_ERROR` en plantillas `{x}` resueltas client-side](next-intl-icu-template-raw.md) — labels que el wizard rellena con `.replace`/`.split` deben leerse con `t.raw`, no `t()`.
- [Una branch-entorno de Neon sin migraciones aplicadas](neon-branch-sin-migraciones.md) — signup anda pero el place no se crea; el auth user queda en el branch del deploy sin schema. Aprovisionar por migraciones (ADR-0017).
- [`getTranslations({locale})` desde page sin `[locale]` en URL](i18n-locale-override-zona-place.md) — la zona "place" bypassa intl middleware; el override de locale (read from DB en S6) funciona como único driver del request config, verificado contra el runtime real.
- [Vercel `modifyConfig` rompe con `path argument undefined` ante combinaciones 4-way de cambios](vercel-modifyconfig-4way-interaction.md) — task #110 demostró por bisección que el adapter `@vercel/next` (Next.js 16.2.6 + Turbopack) falla determinísticamente ante una interacción específica entre runtime export nuevo + archivo de test nuevo + cambios cross-test + import cross-archivo del nuevo export. Local build verde, no reproducible localmente. Workaround: evitar set B (no crear archivo de test nuevo); consolidar tests aunque supere LOC ≤300.
- [El prefijo `__Host-` exige Path=/ y rechaza Domain attribute](host-prefix-cookie-path.md) — cookies `__Host-place_sso_state` y `__Host-place_sso_session` (Feature C SSO) descartadas silently por el browser si Set-Cookie incluye `Domain=` o un `Path` distinto de `/`. RFC 6265 §4.1.3.2; síntoma confuso (handler responde 200 pero la cookie no persiste); tests S3/S4 verifican shape exacto.
- [`PLACE_SSO_SIGNING_KEY` jamás aparece en logs/stdout/error messages](sso-signing-key-no-log.md) — la signing key ES256 (Feature C SSO) en logs = blast radius total. Logging estructurado del flow loggea sólo `{event, code, host, jti, timestamp}`. Tests S2 mockean console y fallan si el output contiene patterns de signing key. Rotación operacional 90d documentada en ADR-0032 §10.
- [jose v6 hardcodea `redirect: 'manual'` en el JWKS fetch y rompe contra apex→www](jose-jwks-redirect-manual.md) — detectado en smoke production T1.1 (Feature C SSO): redeem aterriza en `sso_error=signature_invalid` aunque la firma sea válida porque el JWKS apex `https://place.community/...` responde HTTP 307 a `https://www.place.community/...` (Vercel platform-level) y jose v6 hardcodea `redirect: 'manual'` (`dist/webapi/jwks/remote.js` línea 19) por defensa anti-JWKS-hijack. Fix: `customFetch` Symbol option con policy same-registrable-domain + https + ≤3 hops (ADR-0032 addendum §"Same-registrable-domain redirect policy").
- [Apex login `/[locale]/login` debe honrar `?returnTo` para que cold-start SSO desde custom domain aterrice correctamente](apex-login-returnto-honored.md) — detectado en smoke M1 owner-driven post-S11.2 (Feature C SSO): visitor anónimo en `nocodecompany.co/settings` dispara silent SSO → `sso-issue` redirige a `/[locale]/login?returnTo=<sso-issue URL completa>` correctamente → user loguea → **aterriza en Hub canónico hardcoded en lugar del `returnTo`** porque la page no lee `searchParams.returnTo` (tipo Props no lo incluye) + guard "ya logueado" y `AccessFlow.onSuccess` navegan hardcoded a Hub. Bug pre-existing en login apex; Feature C lo expone sin causarlo (es la primera feature que envía users a `/login?returnTo=…` esperando honor). Fix S11.3 (ADR-0033): helper PURE `validateLoginReturnTo` con allowlist explícito + wire-up minimal en 3 archivos código.
- [Server Action falla con `error`/`none` desde custom domain aunque el user esté autenticado](zone-aware-db-cookie-source.md) — la cookie Neon Auth tiene `Domain=.place.community` (RFC 6265 §5.4 cross-subdomain) → NO llega al custom domain → `requireSessionJwt()` retorna null → `getAuthenticatedDb` directo falla. Síntoma confuso: layout + page funcionan (la cookie SSO local sí está), sólo la mutación falla. Fix canónico (ADR-0034): usar `getAuthenticatedDbForRequest(fn)` (`src/shared/lib/db-for-request.ts`) que detecta `HostZone` + lee la cookie correcta + dispatcha al primitivo apropiado. Invariante going-forward: todo Server Action zone-agnostic usa el coordinator, no el primitivo Feature A directo.
- [Server Action sobre `place_domain` falla `error`/`not_found` para un miembro no-owner](rls-place-domain-owner-only.md) — la policy `place_domain_all` (FOR ALL, USING == WITH CHECK) restringe SELECT/INSERT/UPDATE/DELETE a quien tenga `place_ownership` del place dueño de la fila. Los miembros NO ven `place_domain` (member-read ADR-0021 NO se extiende a config técnica). La policy no tira: filtra silenciosamente y la query devuelve 0 filas afectadas. Invariante: todo nuevo touch a `place_domain` delega al RLS (sin TS gates redundantes); excepción única SECURITY DEFINER `app.lookup_place_by_domain` para el proxy. Cobertura: 18 tests en `rls-place-domain.test.ts` (matriz + structural drift defense).
- [Server Action que llama DEFINER dependiente de `app_user` falla post-signup si no precede con `ensureAppUser` TX 1](accept-invitation-requires-ensure-app-user-tx1.md) — `signUpAccountAction` (ADR-0008 §2/§4) crea SÓLO la identidad Neon Auth, NO siembra `app_user`. El siembrado canon vive en `place-creation/create-place.ts` § bloques `TX 1 — ensureAppUser` + `TX 2 — app.create_place`. Cualquier otro path post-signup hacia un DEFINER que haga lookup en `app_user WHERE auth_user_id = v_auth` rompe con P0002 (`app_user inexistente`). Síntoma confuso: panel muestra "Algo salió mal" genérico + Vercel logs limpios (response 200 envelope, no throw). Fix V1.1 S6 (commit `c13fcfd`) en `acceptInvitationAction`. Pattern obligatorio en cualquier futura action invocable post-signup sin pasar por PlaceWizard.
- [`pnpm test` con ambos projects en paralelo: flake observado una vez, no reproducible](vitest-parallel-projects-flake.md) — 26 fails con WS close en tests de DB el 2026-05-28 (Phase 1.B); investigación 2.C.1 corrió la suite 6x (5 bajo carga 8x, load 65-76) con config idéntica sin reproducirlo → hipótesis de starvation de CPU refutada, causa probable transitoria de Neon. Mitigación higiénica (scripts `test:node`/`test:ui` + CI por pasos), sin tocar pools/concurrency.
- [INSERT/UPDATE/DELETE directos sobre `place_ownership` están denegados — toda mutación pasa por 4 funciones SECURITY DEFINER](place-ownership-defining-functions-only.md) — migration 0012 (Feature D S1, ADR-0035 §3 + §4) implementa WORM-via-DEFINER: REVOKE INSERT/UPDATE/DELETE FROM `app_system` + única policy `po_sel` vía helper `app.current_user_owns_place`. Síntoma de drift = `ERROR: permission denied for table place_ownership` (42501) cuando código de feature intenta SQL directo. Las 4 funciones canónicas (`app.create_place` CU1 / `elevate_to_owner` CU2 / `revoke_ownership` CU3 / `transfer_founder_ownership` CU4) validan invariantes en cuerpo plpgsql + `RAISE EXCEPTION` con P0001 estructural. Defense-in-depth 3 capas: grant SQL + invariantes plpgsql + UNIQUE constraint anti-race. Cobertura: 47 tests TDD en `rls-place-ownership.test.ts` + `elevate-to-owner.test.ts` + `revoke-ownership.test.ts` + `transfer-founder-ownership.test.ts` + `create-place-founder.test.ts` + smoke E2E 5/5 verde contra branch ephemeral Neon.
